# Harness Engineering：用 AI 智能体构建软件

> Sources: OpenAI, 2026-01-28
> Raw: [Harness Engineering（中文）](../../raw/ai-engineering/2026-01-28-harness-engineering.md)

## 概述

OpenAI 用零行人工代码构建并交付了一个内部 beta 产品 — 每一行代码都由 Codex 智能体在五个月内生成，产出约 100 万行代码，通过约 1,500 个 PR 由仅三名工程师推动。核心洞见：当人类不再编写代码时，工程工作从实现转向设计环境、明确意图和构建反馈回路，使智能体能够高效工作。

## 核心约束：不手写代码

团队将"不手写代码"作为核心理念。初始的仓库脚手架、CI 配置，甚至 AGENTS.md 文件本身都由 Codex CLI 使用 GPT-5 生成。该产品拥有真实的内部日常用户和外部 Alpha 测试者 — 并非一个玩具项目。

成果：约 100 万行代码、约 1,500 个 PR、每位工程师每天 3.5 个 PR、耗时约为手工编码的 1/10。

## 重新定义工程师的角色

当智能体承担所有编码工作后，工程师转向：

- **深度优先工作**：将目标拆解为构建模块，提示智能体构建它们，再组合成更复杂的任务。
- **环境设计**：智能体失败时，解决方案几乎不会是"再努力一点" — 而是识别缺失的能力（工具、抽象、文档），并使其对智能体清晰可读且可强制执行。
- **提示驱动交互**：工程师描述任务，智能体打开 PR，智能体对智能体的审查处理了大部分代码审查工作。

> "究竟还需要什么样的能力，我们又该如何让这个能力对智能体来说既清晰可读又可强制执行？"

## 上下文管理：给地图，而非手册

最关键的教训：**给智能体一张地图，而不是一本 1,000 页的说明书。** 庞大指令文件的问题：

- **上下文是稀缺资源**：巨大的指令会挤掉任务相关的信息。
- **过度指导等于没有指导**：当一切都"重要"时，一切都不重要了。
- **立即腐烂**：单一手册会变成陈旧规则的坟场。
- **难以验证**：单一 blob 无法进行机械化的新鲜度/覆盖率检查。

解决方案：AGENTS.md 作为约 100 行的内容目录，指向结构化的 `docs/` 目录。渐进式披露 — 智能体从小处开始，被引导去查看下一步需要的内容。

计划是检入仓库的一等工件。专门的 linter 和 CI 强制执行文档质量。一个"文档园艺"智能体会扫描过时的文档并打开修复 PR。

## 智能体可读性作为设计目标

> "从智能体的角度来看，它在运行时无法在上下文中访问的任何内容都是不存在的。"

仓库必须包含智能体需要的一切 — Google Docs、Slack 线程或人们头脑中的知识对它来说是不可见的。这意味着：

- 将所有上下文推送到版本化的、仓库本地的工件中。
- 倾向于"枯燥的"技术（可组合、API 稳定、在训练集中有良好表现）。
- 有时重新实现功能比绕过不透明的上游库更便宜（例如，自定义 map-with-concurrency 辅助函数代替 `p-limit`，与 OpenTelemetry 集成，100% 测试覆盖率）。

应用可读性同样关键：应用可以按 git worktree 启动，Chrome DevTools 协议接入智能体运行时用于 DOM 截图/导航，本地临时可观测性堆栈向智能体暴露日志（LogQL）和指标（PromQL）。

## 通过不变量强制执行架构

团队对每个业务域强制执行严格的分层架构：

```
Types → Config → Repo → Service → Runtime → UI
```

横切关注点通过单一的 `Providers` 接口进入。其他一切通过自定义 linter 和结构测试机械地强制执行。

关键原则：
- **强制执行不变量，而非实现方式** — 规定什么必须为真（如在边界处解析数据形状），而非如何做。
- **自定义 lint 错误信息包含修复指令**，注入到智能体上下文中。
- **中央边界，地方自主** — 严格的地方必须严格，灵活的地方可以灵活。
- 生成的代码不必符合人类的风格偏好 — 正确性、可维护性和智能体可读性才是标准。

## 熵与垃圾回收

智能体会复制已有的模式 — 包括不够理想的模式。漂移是不可避免的。团队最初每周五（一周的 20%）手动清理"AI 残渣"，但不可扩展。

解决方案：将"黄金原则"编码为机械规则，定期运行后台 Codex 任务扫描偏差、更新质量等级并打开重构 PR。这就像垃圾回收一样 — 以小额增量持续偿还技术债务，而不是让债务累积后再痛苦地一次性偿还。

## 吞吐量改变合并理念

当智能体吞吐量远超人类注意力时：
- 尽量减少阻塞合并门
- PR 生命周期短
- 测试偶发失败通过后续重跑解决，而非无限期阻碍进展
- 纠错成本低，等待成本高

这种权衡是高吞吐量智能体环境特有的，在传统低吞吐量环境中是不负责任的做法。

## 不断提高的自主水平

系统达到了一个门槛：Codex 可以从一个提示端到端驱动一个功能 — 验证代码库状态 → 复现 bug → 录制故障视频 → 实施修复 → 验证修复 → 录制解决视频 → 打开 PR → 回应反馈 → 检测/修复构建故障 → 仅在需要判断时升级给人类 → 合并。

这在很大程度上依赖于特定仓库的结构和工具，不应在没有类似投入的情况下假定可以泛化。

## 待探索的问题

- 在完全由智能体生成的系统中，架构连贯性如何随年份演变？
- 人类判断力在哪些方面能发挥最大杠杆作用，如何编码使其持续发挥作用？
- 随着模型能力不断增强，这一系统将如何演变？

软件工程的纪律现在更多地体现在脚手架上，而非代码上 — 保持代码库一致性的工具、抽象和反馈回路变得越来越重要。
