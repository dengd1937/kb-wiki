过去一年里，我们与数十个跨行业构建大语言模型（LLM）智能体的团队合作过。一致的发现是：最成功的实现并没有使用复杂的框架或专门的库；相反，它们是用简单、可组合的模式构建的。

本文中，我们分享与客户合作以及我们自己构建智能体所学到的经验，并为开发者提供构建高效智能体的实用建议。

## 什么是智能体？

"智能体（Agent）"可以有多种定义。一些客户把智能体定义为：在较长时间内独立运作、使用各种工具完成复杂任务的完全自主系统。另一些人用这个词来描述遵循预定义工作流的、更具规定性的实现。在 Anthropic，我们把所有这些变体都归类为**智能体系统（agentic systems）**，但在**工作流（workflows）**与**智能体（agents）**之间画出一个重要的架构区分：

- **工作流**是通过预定义代码路径来编排 LLM 与工具的系统。
- **智能体**则是 LLM 动态地指导自身流程与工具使用、对"如何完成任务"保持控制权的系统。

下面我们将详细探讨这两类智能体系统。在附录 1（"实践中的智能体"）中，我们描述了客户发现这类系统特别有价值的两个领域。

## 何时（以及何时不）使用智能体

用 LLM 构建应用时，我们建议寻找尽可能最简单的解决方案，只在需要时才增加复杂度。这可能意味着根本不构建智能体系统。智能体系统通常用延迟和成本换取更好的任务表现，你应该考虑这种权衡何时是值得的。

当更高的复杂度是合理的时候，工作流为定义良好的任务提供可预测性与一致性，而智能体则在需要规模化的灵活性与模型驱动决策时是更好的选择。不过对许多应用而言，用检索和上下文示例来优化单次 LLM 调用通常就已足够。

## 何时以及如何使用框架

有许多框架让智能体系统更易实现，包括：

- The Claude Agent SDK；
- AWS 的 Strands Agents SDK；
- Rivet，一个拖放式 GUI 的 LLM 工作流构建器；以及
- Vellum，另一个用于构建与测试复杂工作流的 GUI 工具。

这些框架通过简化调用 LLM、定义与解析工具、把调用串联起来等标准的底层任务，让你容易上手。然而它们常常制造额外的抽象层，可能掩盖底层的提示与响应，使调试更难。它们也容易诱使你在更简单的设置就够用时去增加复杂度。

我们建议开发者从直接使用 LLM API 开始：许多模式只需几行代码就能实现。如果你确实使用框架，请确保理解其底层代码——对"引擎盖下"的错误假设是客户出错的常见来源。

参见我们的 cookbook 获取一些示例实现。

## 构建块、工作流与智能体

本节我们将探讨在生产中见到的智能体系统常见模式。我们从基础构建块——增强型 LLM——开始，逐步增加复杂度，从简单的组合式工作流到自主智能体。

### 构建块：增强型 LLM

智能体系统的基本构建块是一个被检索、工具、记忆等增强能力强化的 LLM。我们当前的模型能主动使用这些能力——生成自己的搜索查询、选择合适的工具、决定保留哪些信息。

![图：增强型 LLM](assets/building-effective-agents/01-augmented-llm.png)

*增强型 LLM（The augmented LLM）*

我们建议聚焦实现中的两个关键方面：把这些能力裁剪到你的具体用例，以及确保它们为你的 LLM 提供一个简单、文档完善的接口。实现这些增强有许多方式，其中一种是通过我们最近发布的 Model Context Protocol，它允许开发者用一个简单的客户端实现就能接入不断增长的第三方工具生态。

本文余下部分，我们将假设每次 LLM 调用都能访问这些增强能力。

### 工作流：提示链（Prompt chaining）

提示链把一个任务分解为一系列步骤，每次 LLM 调用处理上一次的输出。你可以在任何中间步骤上加入程序化检查（见下图中的 "gate"），以确保流程仍在正轨上。

![图：提示链工作流](assets/building-effective-agents/02-prompt-chaining.png)

*提示链工作流（The prompt chaining workflow）*

**何时使用此工作流**：当任务可以被轻松而干净地分解为固定子任务时，此工作流最理想。主要目标是用延迟换取更高的准确率——通过让每次 LLM 调用都成为更容易的任务。

提示链有用的例子：

- 生成营销文案，然后把它翻译成另一种语言。
- 撰写文档大纲，检查大纲是否满足某些标准，然后基于大纲撰写文档。

### 工作流：路由（Routing）

路由对输入进行分类，并将其导向一个专门的后续任务。此工作流允许关注点分离、构建更专门化的提示。没有此工作流，针对一类输入的优化可能损害对其他输入的表现。

![图：路由工作流](assets/building-effective-agents/03-routing.png)

*路由工作流（The routing workflow）*

**何时使用此工作流**：当复杂任务存在更适合分别处理的不同类别，且分类可由 LLM 或更传统的分类模型/算法准确处理时，路由工作得很好。

路由有用的例子：

- 把不同类型的客服查询（一般问题、退款请求、技术支持）导向不同的下游流程、提示与工具。
- 把简单/常见问题路由到更小、更省成本的模型（如 Claude Haiku 4.5），把困难/罕见问题路由到能力更强的模型（如 Claude Sonnet 4.5），以优化整体表现。

### 工作流：并行化（Parallelization）

LLM 有时可以同时处理一个任务，并以程序方式聚合它们的输出。此工作流（并行化）体现为两个关键变体：

- **分段（Sectioning）**：把一个任务拆成并行运行的独立子任务。
- **投票（Voting）**：多次运行同一任务以获得多样化输出。

![图：并行化工作流](assets/building-effective-agents/04-parallelization.png)

*并行化工作流（The parallelization workflow）*

**何时使用此工作流**：当被划分的子任务可并行以提速，或需要多个视角/尝试以获得更高置信度结果时，并行化是有效的。对有多个考量的复杂任务，当每个考量由单独的 LLM 调用处理时，LLM 通常表现更好，因为可对每个具体方面集中注意力。

并行化有用的例子：

- 分段：
  - 实现护栏，一个模型实例处理用户查询，另一个筛查其中不当内容或请求。这往往比让同一次 LLM 调用同时处理护栏与核心响应表现更好。
  - 自动化评测以评估 LLM 表现，每次 LLM 调用评估模型在给定提示上的不同方面。
- 投票：
  - 审查一段代码的漏洞，几个不同的提示审查并在发现问题时标记代码。
  - 评估某段内容是否不当，多个提示评估不同方面，或要求不同的投票阈值以平衡假阳与假阴。

### 工作流：编排者-工作者（Orchestrator-workers）

在编排者-工作者工作流中，一个中央 LLM 动态地分解任务、把它们委派给工作者 LLM，并综合它们的结果。

![图：编排者-工作者工作流](assets/building-effective-agents/05-orchestrator-workers.png)

*编排者-工作者工作流（The orchestrator-workers workflow）*

**何时使用此工作流**：此工作流非常适合无法预测所需子任务的复杂任务（例如在编码中，需要更改的文件数量以及每个文件更改的性质很可能取决于任务本身）。虽然拓扑上相似，但它与并行化的关键区别在于灵活性——子任务不是预先定义的，而是由编排者根据具体输入决定。

编排者-工作者有用的例子：

- 每次都要对多个文件做复杂更改的编码产品。
- 涉及从多个来源收集与分析信息以寻找可能相关信息的搜索任务。

### 工作流：评估者-优化者（Evaluator-optimizer）

在评估者-优化者工作流中，一次 LLM 调用生成响应，另一次在循环中提供评估与反馈。

![图：评估者-优化者工作流](assets/building-effective-agents/06-evaluator-optimizer.png)

*评估者-优化者工作流（The evaluator-optimizer workflow）*

**何时使用此工作流**：当我们有清晰的评估标准，且迭代精炼能带来可衡量价值时，此工作流尤其有效。两个适配信号是：第一，当人类清楚表达反馈时 LLM 响应能被明显改进；第二，LLM 能提供这样的反馈。这类似于人类作者打磨一篇文档时可能经历的迭代写作过程。

评估者-优化者有用的例子：

- 文学翻译，其中存在翻译者 LLM 最初可能捕捉不到的细微之处，而评估者 LLM 能提供有用的批评。
- 需要多轮搜索与分析以收集全面信息的复杂搜索任务，由评估者决定是否需要进一步搜索。

### 智能体（Agents）

随着 LLM 在关键能力上的成熟——理解复杂输入、进行推理与规划、可靠地使用工具、从错误中恢复——智能体正在生产中涌现。智能体的工作始于来自人类用户的命令或与之的交互式讨论。一旦任务明确，智能体独立地规划与运作，可能返回向人类索取进一步信息或判断。在执行期间，智能体在每一步从环境获得"ground truth"（如工具调用结果或代码执行）以评估其进展至关重要。智能体随后可在检查点或遇到阻塞时暂停以获取人类反馈。任务通常在完成时终止，但通常也会加入停止条件（如最大迭代次数）以保持控制。

智能体能处理复杂任务，但其实现往往很直接。它们通常只是 LLM 在一个循环中基于环境反馈使用工具。因此清晰而周到地设计工具集及其文档至关重要。我们在附录 2（"为你的工具做提示工程"）中展开工具开发的最佳实践。

![图：自主智能体](assets/building-effective-agents/07-autonomous-agent.png)

*自主智能体（Autonomous agent）*

**何时使用智能体**：智能体可用于开放式问题——难以或无法预测所需步骤数、且无法硬编码固定路径。LLM 可能运行很多轮，你必须对其决策有一定程度的信任。智能体的自主性使其非常适合在受信任环境中规模化任务。

智能体的自主性意味着更高的成本，以及错误叠加累积的可能。我们建议在沙盒环境中做广泛测试，并配以适当的护栏。

智能体有用的例子（以下来自我们自己的实现）：

- 一个解决 SWE-bench 任务的编码智能体，这些任务涉及根据任务描述编辑多个文件；
- 我们的 "computer use" 参考实现，Claude 使用计算机来完成任务。

![图：编码智能体的高层流程](assets/building-effective-agents/08-coding-agent-flow.png)

## 组合与定制这些模式

这些构建块不是规定性的。它们是开发者可以塑造与组合以适配不同用例的常见模式。成功的关键，与任何 LLM 特性一样，是衡量表现并对实现进行迭代。再强调一次：只有当复杂度能明显改善结果时，才应考虑增加它。

## 总结

在 LLM 领域取得成功，不在于构建最复杂的系统，而在于构建**适合你需求的正确系统**。从简单的提示开始，用全面的评测优化它们，只有当更简单的方案不足时才加入多步智能体系统。

实现智能体时，我们尽量遵循三条核心原则：

- 保持智能体设计的简单。
- 通过显式展示智能体的规划步骤来优先保证透明度。
- 通过充分的工具文档与测试，精心打造你的智能体-计算机接口（ACI）。

框架能帮你快速起步，但当你迈向生产时，不要犹豫去减少抽象层、用基础组件构建。遵循这些原则，你能创造出不仅强大，而且可靠、可维护、被其用户信任的智能体。

## 附录 1：实践中的智能体

我们与客户的合作揭示了 AI 智能体两个特别有前景的应用，它们展示了上述模式的实际价值。两个应用都说明：对于既需要对话又需要行动、有清晰成功标准、能形成反馈回路、并整合有意义人类监督的任务，智能体增加的价值最大。

### A. 客户支持

客户支持把熟悉的聊天机器人界面与通过工具集成增强的能力结合起来。这天然适合更开放式的智能体，因为：

- 支持交互天然遵循对话流，同时需要访问外部信息与行动；
- 工具可以被集成以拉取客户数据、订单历史和知识库文章；
- 退款、更新工单等行动可以程序化处理；以及
- 成功可以通过用户定义的解决方案被清晰衡量。

若干公司已通过"仅对成功解决收费"的按用量计价模式证明了此方法的可行性，显示出对其智能体有效性的信心。

### B. 编码智能体

软件开发领域为 LLM 特性展现了非凡潜力，能力从代码补全演进到自主问题求解。智能体特别有效，因为：

- 代码解决方案可通过自动化测试验证；
- 智能体能以测试结果作反馈对解决方案迭代；
- 问题空间定义良好且结构化；以及
- 输出质量可被客观衡量。

在我们自己的实现中，智能体现在能仅凭 pull request 描述就解决 SWE-bench Verified 基准中的真实 GitHub issue。然而，尽管自动化测试帮助验证功能，人类评审对于确保解决方案符合更广泛的系统需求仍然关键。

## 附录 2：为你的工具做提示工程

无论你构建哪种智能体系统，工具很可能都是你智能体的重要部分。工具通过在我们的 API 中指定其确切结构与定义，使 Claude 能与外部服务和 API 交互。当 Claude 响应时，如果它计划调用某工具，会在 API 响应中包含一个 tool use 块。工具定义与规范应该得到与你整体提示同样多的提示工程关注。在这篇简短的附录中，我们描述如何为你的工具做提示工程。

同一个动作往往有几种指定方式。例如，你可以通过写一个 diff 来指定文件编辑，或通过重写整个文件来指定。对于结构化输出，你可以在 markdown 中或在 JSON 中返回代码。在软件工程中，这类差异是表面的，可以无损地相互转换。然而，某些格式对 LLM 来说比其他格式难写得多。写一个 diff 需要在写新代码之前知道 chunk 头部有多少行在变化；在 JSON 中写代码（相比 markdown）需要额外转义换行与引号。

我们对决定工具格式的建议如下：

- 给模型足够的 token 去"思考"，免得它把自己写进死胡同。
- 让格式接近模型在互联网文本中自然见过的样子。
- 确保没有格式上的"开销"，比如必须对成千上万行代码保持准确计数，或对它写的任何代码做字符串转义。

一条经验法则是：想想人类-计算机接口（HCI）投入了多少精力，并计划在创建良好的智能体-计算机接口（ACI）上投入同样多的精力。以下是一些做法思考：

- 设身处地为模型着想。基于描述与参数，使用这个工具是否显而易见，还是你需要仔细思考？如果是后者，那对模型很可能也是如此。好的工具定义通常包括示例用法、边界情况、输入格式要求，以及与其他工具的清晰边界。
- 你如何更改参数名称或描述以让事情更明显？把这想象成为团队里的初级开发者写一个出色的 docstring。在使用许多相似工具时这尤其重要。
- 测试模型如何使用你的工具：在我们的 workbench 里运行许多示例输入，看看模型犯什么错误，并迭代。
- 对你的工具做防呆（Poka-yoke）。更改参数，使其更难出错。

在为 SWE-bench 构建我们的智能体时，我们实际上花在优化工具上的时间比在整体提示上更多。例如，我们发现当智能体移出根目录后，模型会在使用相对文件路径的工具上出错。为修复这一点，我们把工具改为始终要求绝对文件路径——我们发现模型完美无瑕地使用了这一方法。
